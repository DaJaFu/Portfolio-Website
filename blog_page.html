Building My Portfolio with AWS: Lessons Learned

AWS CloudFront Web Development

Deploying my portfolio on AWS was an incredible learning experience. From S3 buckets to CloudFront distributions, I discovered the intricacies of cloud infrastructure and learned valuable lessons about DNS, caching, and security...

When I decided to build my portfolio from scratch, I knew I wanted to do more than just create a website‚ÄîI wanted to understand the entire deployment pipeline. That's when I chose AWS as my hosting platform.

The Initial Setup

My journey started with creating an S3 bucket. At first, I tried using the S3 static website hosting feature, but I quickly realized this wouldn't give me HTTPS support. That's when I learned about CloudFront and Origin Access Control.

Key Challenges I Faced

  • DNS Configuration: Understanding how Route 53 works and properly configuring A records took some trial and error.
  • SSL Certificates: Learning that certificates must be in us-east-1 for CloudFront was a crucial detail.
  • Cache Invalidation: Figuring out when and how to invalidate CloudFront cache after updates.
  • Origin Access Control: Setting up OAC instead of the legacy OAI method for better security.

What I Learned

The most valuable lesson was understanding the difference between S3 static website hosting and using S3 as a CloudFront origin. While the static website feature is simpler, using CloudFront with OAC provides better security, HTTPS support, and global performance through edge locations.

Technical Architecture

My final setup uses CloudFront as the CDN layer, with an S3 bucket as the private origin. Route 53 handles DNS resolution, and AWS Certificate Manager provides the SSL certificate. This architecture gives me:

  • Global content delivery with low latency
  • HTTPS encryption with custom domain
  • Secure, private S3 bucket accessible only via CloudFront
  • Cost-effective hosting (under $2/month)

Next Steps

Now that the infrastructure is solid, I'm focusing on content optimization‚Äîcompressing images, minifying CSS/JS, and implementing proper cache headers for different file types. I'm also exploring AWS Lambda@Edge for more advanced edge computing capabilities.

If you're considering building your portfolio on AWS, my advice is to start simple with just S3 and CloudFront, then gradually add complexity as you understand each component. The learning curve is steep, but the knowledge you gain is invaluable.

Understanding DNS: From Domain to IP Address

üìÖ January 10, 2025 ‚è±Ô∏è 8 min read üë§ Your Name
DNS Networking Route 53

DNS is often called the "phonebook of the internet," but what does that really mean? Let's dive deep into how DNS works, the different record types, and why understanding this is crucial for any web developer...

When you type a domain name like "google.com" into your browser, your computer doesn't inherently know where to find Google's servers. That's where DNS (Domain Name System) comes in‚Äîit translates human-readable domain names into IP addresses that computers use to communicate.

The DNS Resolution Process

Here's what happens behind the scenes:

  1. Your browser checks its cache for the IP address
  2. If not cached, it asks your operating system
  3. Your OS asks your configured DNS resolver (usually your ISP)
  4. The resolver queries root nameservers
  5. Root servers direct to TLD (Top-Level Domain) nameservers (.com, .org, etc.)
  6. TLD nameservers direct to authoritative nameservers
  7. Authoritative nameservers return the IP address
  8. The IP is cached at multiple levels and returned to your browser

DNS Record Types

Different record types serve different purposes:

  • A Record: Maps domain to IPv4 address
  • AAAA Record: Maps domain to IPv6 address
  • CNAME Record: Creates an alias to another domain
  • MX Record: Directs email to mail servers
  • TXT Record: Stores text information (verification, SPF, etc.)
  • NS Record: Specifies authoritative nameservers

TTL (Time To Live)

TTL determines how long DNS records are cached. A low TTL (like 300 seconds) means changes propagate quickly but generate more DNS queries. A high TTL (like 86400 seconds) improves performance but delays propagation of changes.

Practical Application: My Portfolio

For my portfolio, I use Route 53 with A records (configured as Aliases) pointing to my CloudFront distribution. This gives me:

  • Clean URLs without exposing CloudFront domains
  • Fast DNS resolution through AWS's global network
  • Easy management of DNS records
  • Health checks and failover capabilities (if needed)

Common DNS Issues

During my setup, I encountered several common issues:

  • DNS propagation delays (can take 24-48 hours)
  • Cached DNS records on local machine
  • Incorrect record types (using CNAME for root domain)
  • Nameserver configuration at domain registrar

Understanding DNS has been fundamental to my web development journey. It's one of those "invisible" technologies that just works‚Äîuntil it doesn't, and then you really need to understand what's happening under the hood.

First Week of Coding: Reflections and Progress

Learning Beginner HTML/CSS

Starting my journey into web development has been both exciting and overwhelming. This week, I focused on mastering HTML and CSS fundamentals, and I've learned more than I expected...

I've always been fascinated by how websites work, but I never took the plunge into actually building one until this week. Armed with determination and a text editor, I dove into HTML and CSS.

What I've Learned

The fundamentals are more important than I thought:

  • Semantic HTML creates better structure and accessibility
  • CSS flexbox and grid are game-changers for layouts
  • The box model isn't as confusing as I feared
  • Browser developer tools are incredibly powerful
  • Responsive design requires thinking mobile-first

Challenges Faced

Not everything came easily. I spent hours debugging why my navigation bar wouldn't stay at the top of the page (turns out position: sticky was what I needed). I also struggled with centering elements until I learned about flexbox.

Small Wins

My proudest moment was successfully creating a responsive card layout that looks great on both desktop and mobile. Seeing something I built adapt seamlessly to different screen sizes felt like magic.

What's Next

Next week, I'm diving into JavaScript. I'm excited (and a little nervous) to add interactivity to my projects. I've heard it can be tricky, but I'm ready for the challenge.

If you're just starting out like me, my advice is: don't be afraid to break things. Every error message is a learning opportunity, and every bug you fix makes you a better developer.
